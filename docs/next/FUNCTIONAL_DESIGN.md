# 功能设计

## 核心目标

系统需要围绕一个主目标设计：

- 尽可能找出 MyBatis XML 中真正值得优先处理的慢 SQL

在无法直接使用生产运行时明细的前提下，系统的工作方式调整为：

- 先高召回地找出“高风险慢 SQL 候选”
- 再在测试环境中用 `EXPLAIN`、真实基线执行和结果集基线验证这些候选
- 最后只对已识别出的慢 SQL 做优化和收益验证

## 非目标

- 不以“全量自动修复所有 SQL”为目标
- 不以“直接代表生产真实耗时”作为阶段 3 的承诺
- 不在生产环境直接执行优化 SQL 或 DDL
- 不依赖单一大模型判断慢 SQL

## 关键能力

### 1. 高召回候选发现

- 识别所有 MyBatis statement
- 展开动态 SQL 分支
- 覆盖 `if / choose / foreach / bind / include` 的风险分支
- 对条件字段使用真实分布统计，识别热点值、冷门值、空值、低基数字段、偏斜字段

### 2. 证据化慢 SQL 识别

- 对候选 branch 生成代表性参数 case
- 执行 `EXPLAIN`
- 对优先级更高的 case 做真实基线执行
- 记录计划、耗时、扫描量、返回行数、结果集签名
- 输出明确的 `is_slow / severity / root_causes / impact_score`

### 3. 规则优先的优化生成

- 优先由规则和结构化策略提出优化方案
- LLM 只在具备完整上下文时补充复杂 SQL 改写建议
- 任何优化都必须与阶段 3 的 baseline 建立映射关系

### 4. 优化后验证

- 对优化 SQL 使用同一组参数 case 再次执行
- 验证结果集一致性
- 比较优化前后 `EXPLAIN` 和实际执行基线
- 输出收益、风险与推荐级别

## 核心设计原则

### 高召回优先于低成本

阶段 2 的设计优先保证“不漏掉危险 branch”，再控制执行成本。

### 证据优先于建议

阶段 3 必须先把“它慢不慢、为什么慢”说清楚，阶段 4 才允许给出优化方案。

### 元数据先行

阶段 1 产出的表结构、索引、表行数、字段分布，是后续全部阶段共享的事实基础。

### 分区与分片优先

大项目下任何阶段都不能依赖一个全量聚合 JSON 作为唯一读写入口。

### 稳定 ID

所有阶段必须围绕稳定的 `statement_key / path_id / case_id / finding_id / proposal_id` 建立引用链路。

## 慢 SQL 判定模型

系统不使用单一阈值判定慢 SQL，而采用多信号综合判断：

- 实际执行耗时
- 估算成本
- 扫描行数
- 扫描放大倍数
- 是否全表扫
- 是否发生 filesort / temporary / hash aggregate / nested loop amplification
- 表规模
- 字段选择性
- 结果集返回行数

### 推荐的慢 SQL 判定策略

1. 有真实执行时，以 `actual_time_ms` 为主信号
2. 无真实执行时，以 `EXPLAIN` 风险和扫描放大倍数为主信号
3. 若既有真实执行又有计划风险，则提升置信度

## 排序策略

阶段 5 排名不以 LLM `confidence` 为主，而以 `impact_score` 为主。

建议的 `impact_score` 组成：

- `actual_time_score`: 45%
- `rows_examined_score`: 20%
- `plan_risk_score`: 20%
- `table_scale_score`: 10%
- `result_set_penalty_or_bonus`: 5%

当某 finding 没有真实执行基线时，用 `estimated_cost_score` 替代 `actual_time_score`。

## 测试环境验证策略

由于生产运行时数据不可直接接入，阶段 3 和阶段 4 统一在测试环境执行。

测试环境需要尽量具备：

- 生产对应表结构
- 生产对应索引
- 接近生产的表行数
- 阶段 1 采集的字段分布

如果测试环境不能完整复刻生产数据，可接受“分布保真优先于原值保真”的构造策略。

## 关键输出

系统最终应交付三类结果：

- 已验证慢 SQL
- 高风险但未完全验证的候选
- 每条慢 SQL 的优化方案、收益验证和 patch 方案
