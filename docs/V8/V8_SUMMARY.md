## 二、核心设计决策

| 决策项         | 结论                              | 说明                                      |
|----------------|-----------------------------------|-------------------------------------------|
| **数据格式**   | JSON                              | 保持现状，工具丰富，团队熟悉              |
| **版本管理**   | Git + JSON Schema 混合方案        | schemas/ 存放最新，snapshots/ 存历史      |
| **CLI 设计**   | 双 CLI：sqlopt-cli + sqlopt-data  | 分离优化操作与数据管理职责                |
| **分层存储**   | 项目级缓存 + 运行级产物分离       | 缓存复用，减少重复采集                    |
| **可恢复执行** | Supervisor 状态管理               | 支持中断恢复，状态驱动                    |
| **Java 分析**  | 暂不实施                          | 后期方向，当前聚焦 SQL 优化               |

## 三、7 阶段流程详解

### 阶段 1: Discovery（发现）

**功能**：连接数据库、采集表结构、解析 MyBatis XML

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│ Stage 1: Discovery (发现)                                                          │
│                                                                                     │
│  ┌─────────────────────────────────────────────────────────────────────────────┐   │
│  │ • 连接数据库，验证 DSN                                          [DB] 🐢    │   │
│  │ • 采集表结构、列信息、索引、主键                               [DB] 🐢    │   │
│  │ • 解析 MyBatis Mapper XML                                       [FS] 💾    │   │
│  │ • 提取 SQL 模板（包含 ${} 和 #{} 的未解析 SQL）                  [CPU] ⚡   │   │
│  │ • 识别动态标签（<if>, <choose>, <foreach>）                     [CPU] ⚡   │   │
│  │ • 构建 sqlmap_catalog 索引                                      [CPU] ⚡   │   │
│  └─────────────────────────────────────────────────────────────────────────────┘   │
│                                                                                     │
│  输出: sqlunits.jsonl (SQL单元列表), fragments.jsonl (SQL片段目录)                 │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

**关键组件**：
- `stages/discovery/scanner.py` - XML 扫描器
- `stages/discovery/parser.py` - MyBatis XML 解析器
- `adapters/mybatis_xml.py` - MyBatis XML 适配器

---

### 阶段 2: Branching（分支）

**功能**：将动态 SQL 展开为具体执行分支

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│ Stage 2: Branching (分支)                                                         │
│                                                                                     │
│  ┌─────────────────────────────────────────────────────────────────────────────┐   │
│  │ • 解析 SqlNode 树，构建分支图                                    [CPU] ⚡   │   │
│  │ • 根据策略展开分支:                                                        │   │
│  │   - all_combinations: 所有条件组合                                    │   │
│  │   - pairwise: 成对组合（两两组合）                                     │   │
│  │   - boundary: 边界值组合                                               │   │
│  │ • 互斥分支检测，避免冗余展开                                [CPU] ⚡    │   │
│  │ • 分支条件记录与追踪                                        [CPU] ⚡    │   │
│  └─────────────────────────────────────────────────────────────────────────────┘   │
│                                                                                     │
│  输出: branches.jsonl (分支列表)                                                   │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

**关键组件**：
- `stages/branching/branch_generator.py` - 分支生成器
- `stages/branching/sql_node.py` - SQL 节点树
- `stages/branching/branch_strategy.py` - 分支策略
- `stages/branching/mutex_branch_detector.py` - 互斥分支检测

---

### 阶段 3: Pruning（剪枝）

**功能**：静态分析、风险标记、低价值分支过滤

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│ Stage 3: Pruning (剪枝)                                                           │
│                                                                                     │
│  ┌─────────────────────────────────────────────────────────────────────────────┐   │
│  │ • 静态分析每个分支 SQL                                       [CPU] ⚡        │   │
│  │ • 风险检测:                                                              │   │
│  │   - prefix_wildcard: LIKE '%value'                           [HIGH]      │   │
│  │   - suffix_wildcard_only: LIKE 'value%'                       [LOW]       │   │
│  │   - concat_wildcard: CONCAT('%', value)                       [HIGH]      │   │
│  │   - function_wrap: UPPER(col) = ?                             [MEDIUM]    │   │
│  │   - select_star: SELECT *                                     [MEDIUM]    │   │
│  │   - missing_limit: 无 LIMIT 子句                              [MEDIUM]    │   │
│  │ • 分支聚合: 去重 + 归类                                      [CPU] ⚡    │   │
│  │ • 剪枝: 过滤低风险分支                                       [CPU] ⚡    │   │
│  └─────────────────────────────────────────────────────────────────────────────┘   │
│                                                                                     │
│  输出: risks.jsonl (风险列表)                                                      │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

**风险等级**：
| 标记                  | 模式                  | 风险等级 |
|-----------------------|----------------------|----------|
| prefix_wildcard      | `'%'+name+'%'`       | HIGH     |
| suffix_wildcard_only | `name+'%'`           | LOW      |
| concat_wildcard      | `CONCAT('%',name)`   | HIGH     |
| function_wrap         | `UPPER(name)`        | MEDIUM   |

## 四、CLI 工具

### 4.1 sqlopt-cli（SQL 优化）

```bash
# 验证配置
sqlopt-cli validate-config --config sqlopt.yml

# 执行完整流程
sqlopt-cli run --config sqlopt.yml

# 查看状态
sqlopt-cli status --run-id <run_id>

# 恢复中断的运行
sqlopt-cli resume --run-id <run_id>

# 应用补丁
sqlopt-cli apply --run-id <run_id>

# 诊断模式（只执行阶段 1-3）
sqlopt-cli diagnose --config sqlopt.yml

# 单独执行某个阶段
sqlopt-cli optimize --run-id <run_id> --sql-key <sql_key>
sqlopt-cli validate --run-id <run_id> --sql-key <sql_key>
```

**对应阶段**：
| 命令          | 执行阶段     |
|--------------|-------------|
| `run`        | 1-7 全部    |
| `diagnose`   | 1-3         |
| `optimize`   | 5           |
| `validate`   | 6           |
| `apply`      | 7           |

## 五、数据目录设计

### 5.1 分层结构

```
项目根目录/
│
├── .sqlopt/                          # 项目级持久存储
│   ├── cache/
│   │   ├── db_schemas/              # 表结构缓存（按DB哈希）
│   │   └── sqlmap_cache/            # SQL模板缓存
│   └── history/                      # 历史运行归档
│
├── runs/                            # 运行时产物
│   └── {run_id}/
│       ├── project_context/          # 全局上下文
│       │   └── meta.json
│       ├── sqlmap_catalog/          # SQL目录
│       │   ├── index.json          # 索引文件
│       │   └── {sql_key}.json     # 单个SQL详情
│       ├── branches/                # 分支数据
│       │   └── {sql_key}.json
│       ├── risks/                   # 风险数据
│       │   └── {sql_key}.json
│       ├── baseline/               # 基线数据
│       │   └── {sql_key}.json
│       ├── proposals/               # 优化建议
│       │   └── {sql_key}/
│       │       ├── prompt.json
│       │       └── proposal.json
│       ├── acceptance/             # 验证结果
│       │   └── {sql_key}.json
│       ├── patches/               # 补丁文件
│       │   └── {sql_key}/
│       │       └── patch.xml
│       └── supervisor/            # 运行状态
│           ├── meta.json
│           ├── state.json
│           └── plan.json
│
└── contracts/                      # 数据契约 Schema
    ├── sqlunit.schema.json
    ├── fragment_record.schema.json
    ├── optimization_proposal.schema.json
    ├── acceptance_result.schema.json
    └── patch_result.schema.json
```

### 5.2 设计原则

**分层原则**：
- **项目级缓存**：可被多次运行复用，减少重复采集
- **运行级产物**：本次运行的完整数据，独立隔离

**缓存策略**：
- 表结构缓存：按数据库连接哈希存储，支持 TTL 过期
- SQL 模板缓存：按 XML 文件 hash 缓存，文件变化自动失效

**加载策略**：
- 索引快速加载：仅加载 index.json 即可列出所有 SQL
- 详情按需加载：需要查看详情时再加载具体文件
- 阶段批量加载：按阶段目录批量加载进行全量分析

## 六、版本管理

### 6.1 契约版本管理

```
contracts/
├── schemas/                        # 主 Schema（最新版本）
│   ├── sqlunit.schema.json
│   ├── fragment_record.schema.json
│   └── ...
├── snapshots/                      # 历史版本快照
│   ├── v1.0.0/
│   ├── v1.1.0/
│   └── v1.2.0/
├── versions.json                  # 版本索引
└── CHANGELOG.md                  # 变更日志
```

### 6.2 语义化版本

| 版本类型 | 规则                                      |
|---------|------------------------------------------|
| MAJOR   | 不兼容的结构变更                          |
| MINOR   | 新增可选字段，向后兼容                    |
| PATCH   | 文档格式修正                              |

### 6.3 版本元数据设计

每个 Schema 文件包含版本元数据：
- `x-version`：版本号和发布日期
- `x-addedIn`：字段添加版本
- `x-default`：默认值
- `x-compatibility`：向前/向后兼容声明

## 九、术语表

| 术语          | 定义                                                    |
|--------------|--------------------------------------------------------|
| SQL 模板      | 包含 ${} 和 #{} 的未解析 SQL                            |

| 风险等级      | high / medium / low                                    |
| 基线          | SQL 执行时的性能数据                                     |
| sqlmap_catalog| 每个 Mapper Method 的完整信息目录                        |
| SqlNode       | SQL 节点树，描述动态 SQL 结构                           |
| 互斥分支      | 同一时刻只会执行一个的分支路径                           |
