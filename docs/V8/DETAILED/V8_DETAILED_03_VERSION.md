# 详细设计 3：版本管理方案

## 1. 设计目标

建立数据契约的版本管理体系，确保向前/向后兼容，支持版本追溯。

## 2. 方案对比

### 2.1 方案 A：纯 Git 管理

- 版本历史完整
- 运行时切换不灵活

### 2.2 方案 B：JSON 内置版本

- 运行时可动态选择
- 版本历史需要额外记录

### 2.3 方案 C：混合方案（推荐）

结合两者优点：
- 使用 Git 管理版本历史
- 使用 JSON 扩展存储元数据

## 3. 目录结构

```
contracts/
├── schemas/                    # 主 Schema 目录（最新版本）
├── snapshots/                  # 历史版本快照
│   ├── v1.0.0/
│   ├── v1.1.0/
│   └── v1.2.0/
├── versions.json              # 版本元数据索引
└── CHANGELOG.md             # 变更日志
```

## 4. 设计思路

### 4.1 目录分工

- **schemas/**：始终保持最新版本，开发时直接修改
- **snapshots/**：发布新版本时自动生成快照
- **versions.json**：版本索引，记录版本与契约的映射关系
- **CHANGELOG.md**：详细变更记录

### 4.2 语义化版本

```
MAJOR.MINOR.PATCH
  │     │     └── 文档格式修正
  │     └──────── 新增可选字段
  └────────────── 不兼容变更
```

### 4.3 CLI 命令

- `contracts version`：查看当前版本
- `contracts diff --from v1.0.0 --to v1.1.0`：对比差异
- `contracts publish --version v1.3.0 --changes "新增..."`：发布新版本
