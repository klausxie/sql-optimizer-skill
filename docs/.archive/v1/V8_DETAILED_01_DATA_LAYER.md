# 详细设计 1：分层数据目录设计

## 1. 设计目标

解决两个核心问题：
1. 数据膨胀：每次运行产生的 JSON 文件越来越大
2. 复用需求：同一数据库的表结构可能被多次采集

## 2. 分层结构设计

### 2.1 整体架构

```
项目根目录/
│
├── .sqlopt/                        # 项目级持久存储
│   ├── cache/                     # 缓存（可复用）
│   │   ├── db_schemas/          # 按数据库连接区分的表结构缓存
│   │   │   └── {db_hash}/      # DB 连接哈希作为目录名
│   │   │       ├── _meta.json  # 采集时间、DB 版本
│   │   │       └── tables/
│   │   └── sqlmap_cache/        # 解析后的 SQL 模板缓存
│   │
│   └── history/                   # 历史运行归档
│
├── runs/                         # 运行时产物
│   └── {run_id}/
│       ├── project_context/       # 全局上下文（本次运行）
│       ├── sqlmap_catalog/      # SQL 目录
│       └── supervisor/
│
└── contracts/                    # 数据契约 Schema
```

### 2.2 各层职责

| 层级 | 位置 | 内容 | 生命周期 |
|------|------|------|----------|
| 缓存层 | `.sqlopt/cache/` | 表结构、SQL 模板 | 可被多个 run 复用 |
| 运行层 | `runs/<run_id>/` | 本次运行的全部产物 | 本次运行 |

## 3. 缓存策略

### 3.1 表结构缓存

**缓存键生成**：
- 使用数据库连接 DSN 的哈希值作为缓存目录名
- 同一数据库只需采集一次表结构

**缓存元数据**：
- 采集时间、数据库版本、表数量
- 过期时间（TTL）

### 3.2 缓存使用逻辑

1. 运行开始时，检查缓存是否存在且未过期
2. 缓存命中：直接使用缓存的表结构
3. 缓存未命中：从数据库重新采集并写入缓存

### 3.3 缓存失效

- 手动失效：用户主动清除缓存
- TTL 过期：超过设定时间自动失效
- 主动失效：检测到数据库 Schema 变化

## 4. 运行层设计

### 4.1 Project Context（项目上下文）

本次运行采集的全局信息：
- 项目元信息
- 数据库连接信息
- 表结构
- 索引信息

### 4.2 Sqlmap Catalog（SQL 目录）

每个 Mapper Method 的完整信息：
- 索引：快速定位所有 SQL
- 详情：单个 SQL 的完整信息
- 分支：分支展开结果
- 基线：性能基线数据
- 提案：优化建议
- 验证：验证结果

## 5. 加载策略

### 5.1 场景化加载

| 场景 | 加载方式 |
|------|----------|
| 列表展示 | 仅加载 sqlmap_catalog/index.json |
| 详情查看 | 加载 index.json + 详情 |
| 全量分析 | 批量加载需要的阶段 |
| 归档清理 | 按阶段删除历史数据 |

### 5.2 设计优势

- 索引快速定位：O(1) 找到目标
- 按需加载：内存占用降低
- 阶段隔离：可单独删除历史数据
- 并行读取：多文件可并行加载
