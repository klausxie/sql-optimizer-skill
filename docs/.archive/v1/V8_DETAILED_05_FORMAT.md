# 详细设计 5：数据格式选择

## 1. 格式对比

| 格式 | 人类可读 | 体积 | 工具支持 | 适用场景 |
|------|----------|------|----------|----------|
| JSON | ★★★★☆ | 中 | ★★★★★ | 通用数据交换 |
| YAML | ★★★★★ | 中 | ★★★★☆ | 配置文件 |
| TOML | ★★★★★ | 小 | ★★★☆☆ | 简单配置 |
| MessagePack | ★☆☆☆☆ | 小 | ★★★☆☆ | 紧凑传输 |
| Protobuf | ★☆☆☆☆ | 最小 | ★★★★★ | 高效序列化 |

## 2. 场景化选择

| 场景 | 推荐格式 | 理由 |
|------|----------|------|
| 用户配置文件 | YAML | 可读、易写、支持注释 |
| 数据契约 Schema | JSON Schema | 验证能力强 |
| 运行产物 | JSON/JSONL | 工具兼容 |
| 内部传输 | MessagePack | 性能优先 |
| API 响应 | JSON | 通用标准 |

## 3. 当前选择

**继续使用 JSON**

理由：
- 现状良好，工作正常
- 工具丰富
- 团队熟悉

**后续可改进**：
- 配置文件 → YAML（需要更多注释时）
- 大文件 → MessagePack（数据量变大时）
