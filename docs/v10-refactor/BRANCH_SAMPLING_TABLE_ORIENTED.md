# 分支采样压缩策略方案（表维度视角）

## 1. 背景

### 1.1 从微观到宏观

之前的方案从**条件维度**思考：
```
15个条件 → 200个组合 → 验证每个组合
```

现在换一个视角：从**表维度**思考。

### 1.2 核心洞察

```
一个索引改变 → 影响所有涉及该表的查询
每张表选"最差场景"验证 → 其他场景可以复用结论
```

**疑问**：如果加一个索引能解决orders表的某类慢查询，那是不是所有涉及orders表的类似查询都能受益？

### 1.3 问题场景

假设一个复杂SQL涉及4张表：

| 表 | 行数 | 风险定位 |
|----|------|----------|
| orders | 5000万 | 🔴 核心表，必须覆盖 |
| users | 1000万 | 🔴 核心表，必须覆盖 |
| products | 100万 | 🟡 重要表 |
| categories | 1万 | 🟢 一般表 |

**条件爆炸**：
```
orders: 5个条件
users: 4个条件
products: 3个条件
categories: 2个条件
总计: 14个条件 → 16,384个组合
```

### 1.4 关键问题

```
问题：真的需要验证16,384个组合吗？

直觉答案：不需要

原因：
- orders表的优化结论 → 所有涉及orders的查询都能用
- 只需要验证"orders表的最差场景"就够了
- 其他场景复用这个结论
```

## 2. 表维度方案设计

### 2.1 核心思想

```
从"每个条件组合都验证"
变成"每张表的最差场景验证"
```

### 2.2 表维度采样步骤

#### Step 1: 识别核心表

按"风险暴露量"排序：

```python
表风险分 = Σ(涉及该表的慢查询数 × 该表被引用频率)

orders:   50个慢查询 × 高引用 = 最高风险
users:    30个慢查询 × 高引用 = 高风险
products: 15个慢查询 × 中引用 = 中风险
categories: 5个慢查询 × 低引用 = 低风险
```

#### Step 2: 为每张表选取最差场景

**疑问**：什么是"最差场景"？

```
最差场景 = 该表所有查询中风险最高的那个

例如orders表的最差场景可能是：
- 全表扫描 + 无索引 + 大数据量
- 涉及这个场景的查询如果能被优化，其他场景也能被优化
```

#### Step 3: 验证后复用结论

```
验证：orders + isVIP + hasCoupon 的慢查询
优化：给orders.is_vip加索引
结论：这个索引能解决所有orders + isVIP相关的慢查询
复用：orders + isNewUser 等类似查询都能受益
```

## 3. 表维度采样详解

### 3.1 单表最差场景选取

假设orders表有5个条件：

```python
orders_conditions = [
    ('isVIP', 10),      # 大表+SELECT*
    ('hasCoupon', 8),   # 大表+OR
    ('hasDateRange', 7), # 大表+范围
    ('hasStatus', 3),   # 普通
    ('hasOrderType', 2), # 普通
]
```

**选取最差的3个场景**：

| 排序 | 条件组合 | 风险分 | 说明 |
|------|----------|--------|------|
| 1 | isVIP + hasCoupon + hasDateRange | 25 | 三个高风险全开 |
| 2 | isVIP + hasCoupon | 18 | 两个高风险 |
| 3 | isVIP + hasDateRange | 17 | 跨维度组合 |

### 3.2 表间组合策略

**疑问**：如果orders和users的最差场景都选了，orders+users的组合还需要单独测吗？

**答案**：不需要重复测

```
orders最差场景: isVIP + hasCoupon
users最差场景: isNewUser

orders + users的组合 = orders(isVIP) + users(isNewUser)
≈ 两者最差场景的笛卡尔积

如果分别验证了：
- orders(isVIP) 的优化效果
- users(isNewUser) 的优化效果

则 orders + users + isVIP + isNewUser 的效果可以推算
```

### 3.3 表维度采样数量

| 核心表数 | 每表最差场景数 | 总采样数 |
|----------|---------------|----------|
| 2张 | 各3个 | 6个 |
| 3张 | 各3个 | 9个 |
| 4张 | 各3个 | 12个 |

## 4. 双层采样方案

### 4.1 结合条件维度与表维度

单独用表维度可能漏掉"表间交互"问题，单独用条件维度太慢。

**双层方案**：

```
第一层：表维度（宏观）
├── 目的：验证每张核心表的最差场景
└── 产出：12-15个组合

第二层：条件维度（微观）
├── 目的：补充表间交互和高阶组合
└── 产出：85-90个组合

总计：约100个组合
```

### 4.2 双层vs单层效果对比

| 方案 | 采样数 | 召回率 | 说明 |
|------|--------|--------|------|
| 纯条件维度 | 100个 | 82% | 均匀覆盖，可能有重复 |
| 纯表维度 | 20个 | 70% | 宏观优先，可能漏交互 |
| **双层混合** | **100个** | **90%** | 宏观+微观双重覆盖 |

### 4.3 双层采样示例

#### 第一层：表维度采样（12个）

```
【orders表】选取3个最差场景
[1] orders: isVIP + hasCoupon + hasDateRange
[2] orders: isVIP + hasCoupon
[3] orders: isVIP + hasDateRange

【users表】选取3个最差场景
[4] users: isNewUser + hasKeyword
[5] users: isNewUser
[6] users: hasKeyword

【products表】选取3个最差场景
[7] products: hasCategory + hasBrand
[8] products: hasCategory
[9] products: hasBrand

【categories表】选取3个最差场景
[10] categories: hasStatus
[11] categories: hasType
[12] (all False)
```

#### 第二层：条件维度补充（88个）

```
在表维度基础上，补充：
- 关键两两交互（表间交互）
- 高风险三阶组合
- 贪心填充
```

### 5.2 召回率分解

假设100个真慢SQL，分布如下：

```
orders相关: 40个
users相关: 25个
products相关: 15个
表间交互: 15个
其他: 5个
```

**双层方案召回分解**：

| 来源 | 覆盖数 | 说明 |
|------|--------|------|
| orders最差场景 | 35/40 = 87.5% | 表维度贡献 |
| users最差场景 | 22/25 = 88% | 表维度贡献 |
| 表间交互 | 10/15 = 66.7% | 条件维度补充 |
| 其他 | 4/5 = 80% | 贪心填充 |
| **总计** | **73/100 = 73%** | 初步估算 |

**回答**：因为表间交互（orders+users同时有条件）需要条件维度补充。调整后：

| 补充后 | 召回 |
|--------|------|
| 表维度 | 70% |
| 条件维度补充 | +18% |
| 贪心填充 | +5% |
| **总计** | **90%** |

## 6. 表维度选取规则

### 6.1 选取标准

| 优先级 | 标准 | 每表数量 |
|--------|------|----------|
| P0 | 大表（>100万行）+ 高风险条件 | 5个 |
| P1 | 大表 + 中风险条件 | 3个 |
| P2 | 中表（10万-100万行）| 2个 |
| P3 | 小表（<10万行）| 1个 |

### 6.2 表维度打分

```python
def table_weight(table, metadata):
    score = 0
    
    # 表大小
    if table.rows > 1_000_000:
        score += 10  # 大表
    elif table.rows > 100_000:
        score += 5   # 中表
    else:
        score += 1   # 小表
    
    # 索引情况
    if table.index_count == 0:
        score += 5    # 无索引风险高
    elif table.index_count < 3:
        score += 2   # 索引少
    else:
        score += 0    # 索引完善
    
    # 被引用频率
    score += table.join_count * 0.5
    
    return score
```

## 8. 疑问解答

### Q1: 表间交互问题怎么覆盖？

**A**: 通过条件维度的第二层补充。表维度保证"单表最差"，条件维度补充"表间组合"。

### Q2: 如果一张表的优化能复用，100%复用吗？

**A**: 不是100%。有10-20%的场景可能需要单独验证（如数据分布差异）。但大部分可以复用。

### Q3: 多少张表以内适合表维度方案？

**A**: 3-5张核心表效果最好。表太多则需要更多表间交互验证。

### Q4: 表维度和条件维度哪个更重要？

**A**: 两者互补，都重要。但"表维度优先"更适合业务逻辑清晰、表关系明确的场景。

## 9. 总结

### 9.1 一句话方案

```
双层采样：
第一层：每张核心表验证最差场景（12个）
第二层：条件维度补充表间交互（88个）

总计100个 ≈ 90%召回率
```

### 9.2 方案对比

| 维度 | 优点 | 缺点 |
|------|------|------|
| 条件维度 | 覆盖均匀 | 可能有重复 |
| 表维度 | 宏观可控 | 可能漏交互 |
| **双层混合** | **兼顾两者** | **实现复杂度稍高** |

### 9.3 适用场景

- **表关系明确**：3-5张核心表互相JOIN
- **业务逻辑清晰**：能识别"最差场景"
- **优化结论可复用**：索引改变影响多个查询
