# 分支采样压缩策略方案

## 1. 背景

### 1.1 问题场景

当SQL包含多个独立动态条件（`<if>`标签）时，组合数量呈指数级增长：

| 条件数 | 理论组合数 | 可控规模 |
|--------|------------|----------|
| 5 | 32 | ✅ 可接受 |
| 8 | 256 | ⚠️ 偏高 |
| 10 | 1,024 | ❌ 爆炸 |
| 15 | 32,768 | ❌❌ 无法处理 |

### 1.2 核心问题

如何从数万种组合中，选取**少量代表性组合**进行优化，同时**最大化覆盖真正需要优化的慢SQL**？

### 1.3 假设说明

**为什么用"100个真慢SQL"做比喻？**

实际情况中，无法精确知道展开后有多少"真慢SQL"，但基于业务经验可以合理假设：

```
假设：100个真慢SQL

问题分布：
- 60% 单因子问题（某个条件单独就能导致慢）
- 30% 两两交互问题（两个条件组合才导致慢）
- 10% 三阶及以上问题（三个及以上条件组合，较少见）

结论：
- 单因子+两两交互 ≈ 90%的问题
- 三阶及以上问题很少
- 因此100-200个采样大概能解决70-90%的问题
```

> 注：这里的"100个"和比例是经验假设，实际需要根据业务场景调整。

---

## 2. 方案设计

### 2.1 核心思想

采用**阶梯式贪心采样**：

```
Step 1: 单因子覆盖      → 保证每个条件的开关都被覆盖
Step 2: 高权重两两交互  → 保证高风险的条件对被覆盖
Step 3: 高权重三阶交互  → 补充高阶风险
Step 4: 贪心得分填充    → 按风险权重补满预算
```

### 2.2 风险权重体系

#### 基础分（SQL文本特征）

| 特征 | 分值 | 说明 |
|------|------|------|
| 涉及大表 | +3 | users/orders/logs等 |
| 包含JOIN | +2 | JOIN是性能杀手 |
| SELECT * | +2 | 回表开销 |
| 包含子查询 | +2 | 子查询放大数据量 |
| LIKE前缀通配 | +2 | `LIKE '%xxx'` 无法用索引 |
| OR条件 | +1 | OR难以使用索引 |
| ORDER BY/GROUP BY | +1 | 排序分组有成本 |

#### 元数据调整系数

| 维度 | 系数 | 说明 |
|------|------|------|
| 表大小 | ×10~0.1 | 大表扫描成本高 |
| 索引覆盖 | ×0.1~1.0 | 有索引大幅降风险 |
| 字段类型 | ×0.8~1.5 | 大字段增加成本 |

#### 最终权重

```
最终权重 = 基础分 × 表大系数 × 索引系数
```

## 3. 阶梯方案详解

### 3.1 Step 1: 单因子覆盖

**目标**：保证每个条件的T/F状态至少出现一次

**原理**：大部分SQL问题来自"某个条件导致全表扫描"，单因子覆盖能100%捕获这类问题。

**示例**：15个条件

```
组合1: T F F F F F F F F F F F F F F   (条件1=T)
组合2: F T F F F F F F F F F F F F F   (条件2=T)
组合3: F F T F F F F F F F F F F F F   (条件3=T)
...
组合15: F F F F F F F F F F F F F F T  (条件15=T)
组合16: F F F F F F F F F F F F F F F  (all False)
```

**产出**：16个组合

---

### 3.2 Step 2: 高权重两两交互

**目标**：覆盖高风险的条件组合

**原理**：两个条件组合可能产生新的性能问题（如索引同时失效）。

**示例**：

| 排序 | 条件对 | 权重和 | 说明 |
|------|--------|--------|------|
| 1 | hasKeyword + isVIP | 180+140=320 | 大表+无索引+LIKE |
| 2 | hasKeyword + hasCoupon | 180+112=292 | 大表+无索引+OR |
| 3 | isVIP + hasCoupon | 140+112=252 | 大表+双条件 |
| ... | ... | ... | ... |

**产出**：覆盖top-10对，每对1-2个组合，约15-20个新增组合

---

### 3.3 Step 3: 高权重三阶交互

**目标**：补充三阶高风险组合

**原理**：三阶问题较少，但高权重三阶（如三个大表条件同时为T）仍需覆盖。

**示例**：

| 排序 | 条件组合 | 权重和 | 说明 |
|------|----------|--------|------|
| 1 | hasKeyword + isVIP + hasCoupon | 432 | 三个大表无索引条件 |
| 2 | hasKeyword + isVIP + hasDateRange | 390 | 大表+LIKE+范围 |
| ... | ... | ... | ... |

**产出**：覆盖top-5三阶组合，约5个新增组合

---

### 3.4 Step 4: 贪心得分填充

**目标**：按风险权重贪心选取，补满预算

**原理**：剩余预算给最高风险组合。

**计算**：
```
组合风险得分 = Σ(条件的最终权重)
```

**示例**：得分最高的待选组合

```
hasKeyword + isVIP + hasCoupon + hasDateRange: 432分
hasKeyword + isVIP + hasCoupon: 432分
...
```

**产出**：补满到目标预算（如50/100/200个）

## 4. 完整方案效果

### 4.1 采样规模与召回率

基于假设"100个真慢SQL"估算：

| 预算 | 单因子 | 两两交互 | 三阶交互 | 覆盖真慢SQL | 召回率 |
|------|--------|----------|----------|------------|--------|
| **50个** | 16个(100%) | ~10对(20%) | ~3个 | ~62个 | **62%** |
| **100个** | 16个(100%) | ~30对(55%) | ~10个 | ~82个 | **82%** |
| **200个** | 16个(100%) | ~60对(85%) | ~25个 | ~92个 | **92%** |
| **300个** | 16个(100%) | 105对(100%) | ~50个 | ~96个 | **96%** |

> 注：以上为经验估算，实际效果需验证

### 4.2 各规模适用场景

| 预算 | 适用场景 | 召回率 | 说明 |
|------|----------|--------|------|
| **50个** | 快速验证 | ~62% | 快速发现问题，适合开发调试 |
| **100个** | 均衡模式 | ~82% | 平衡覆盖与成本，推荐日常使用 |
| **200个** | 深度分析 | ~92% | 追求全面覆盖，适合上线前检查 |
| **300个** | 极限覆盖 | ~96% | 最高覆盖，适合重要SQL深度优化 |

### 4.3 边际收益分析

```
50 → 100个：+20pp召回（+100%成本）
100 → 200个：+10pp召回（+100%成本）
200 → 300个：+4pp召回（+50%成本）

结论：100-200个是甜点区间，性价比最高
```

## 5. 元数据增强

### 5.1 为什么加元数据？

**目的**：让采样更精准，优先选取"真正高风险"的组合

| 无元数据 | 有元数据 |
|----------|----------|
| 所有大表条件同等对待 | 区分"有索引"和"无索引" |
| 小表风险可能被高估 | 小表正确降权 |
| 采样中有较多"假高风险" | 采样更精准 |

### 5.2 元数据调整示例

```python
# 条件: hasKeyword = 大表orders + LIKE '%xxx%' + 无索引
基础分: 9
表大系数: ×10 (大表)
索因系数: ×1.0 (无索引)
最终权重: 9 × 10 × 1.0 = 90分

# 条件: hasColor = 小表products + 有索引
基础分: 2
表大系数: ×0.1 (小表)
索因系数: ×0.3 (有索引)
最终权重: 2 × 0.1 × 0.3 = 0.06分
```

### 5.3 元数据增强效果

| 预算 | 无元数据召回 | 有元数据召回 | 提升 |
|------|-------------|-------------|------|
| 50个 | 62% | 72% | +10pp |
| 100个 | 82% | 88% | +6pp |
| 200个 | 92% | 94% | +2pp |

## 6. 配置项设计

```yaml
parse:
  # 分支采样配置
  branch_sampling:
    # 是否启用采样（关闭则全展开）
    enabled: true
    
    # 采样预算
    budget: 100  # 可选50/100/200/300
    
    # 阶梯配置
    ladder:
      single_factor: true      # Step1: 单因子覆盖
      pairwise: true          # Step2: 两两交互
      triple: true            # Step3: 三阶交互
      greedy: true            # Step4: 贪心填充
    
    # 元数据增强
    metadata_boost:
      enabled: true
      table_size_boost: true
      index_coverage_boost: true
    
    # 回退策略
    fallback: "full_expand"  # 组合数<预算时：full_expand | greedy
```

## 7. 实施建议

### 7.1 推荐流程

```
1. 先用50个快速验证 → 发现问题多
2. 扩容到100个深度分析 → 大部分问题已覆盖
3. 重要SQL用200个 → 上线前确保无遗漏
```

### 7.2 自适应策略

Recog阶段发现高风险但未采样的组合 → 自动补充采样 → 迭代优化

### 7.3 长期迭代

积累足够数据后，根据实际召回率调整各阶梯的采样比例，找到最优配置。

---

## 8. 总结

### 8.1 一句话方案

```
阶梯式贪心采样：
单因子100%覆盖 + 两两交互按权重覆盖 + 贪心填充补预算

100个采样 ≈ 80%召回
200个采样 ≈ 92%召回
```

### 8.2 核心价值

| 价值 | 说明 |
|------|------|
| **可控** | 预算固定，不会指数爆炸 |
| **可解释** | 每一步采样都有明确理由 |
| **可调节** | 各阶梯参数可配置 |
| **可迭代** | 支持自适应补充采样 |
