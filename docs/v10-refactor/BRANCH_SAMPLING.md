# 分支采样压缩策略方案

## 1. 问题背景

### 1.1 分支爆炸场景

当SQL包含多个独立动态条件（`<if>`标签）时，组合数量呈指数级增长：

| 条件数 | 理论组合数 | 可控规模 |
|--------|------------|----------|
| 5 | 32 | ✅ 可接受 |
| 8 | 256 | ⚠️ 偏高 |
| 10 | 1024 | ❌ 爆炸 |
| 15 | 32768 | ❌❌ 无法处理 |

### 1.2 目标

在可控规模内（如50-100个组合），尽可能高地覆盖可能存在性能问题的分支。

---

## 2. 核心概念

### 2.1 单因子效应

**定义**：只改变一个条件，其他条件保持不变，观察该条件"开/关"对性能的影响。

**示例**：10个if条件 `a,b,c,d,e,f,g,h,i,j`

```
组合1: T F F F F F F F F F  (只有a开启)
组合2: F T F F F F F F F F  (只有b开启)
组合3: F F T F F F F F F F  (只有c开启)
...
组合11: F F F F F F F F F F (全部关闭=default)
```

**作用**：捕捉"某个条件单独就能导致性能问题"的场景（如`a=true`时必定全表扫描）。

---

### 2.2 两两交互

**定义**：两个条件同时为true时的组合效应。

**示例**：
```
组合: T T F F F F F F F F  (a=T, b=T)
```

**作用**：捕捉"两个条件组合才触发问题"的场景（如`a=true AND b=true`时走嵌套循环）。

---

### 2.3 高阶交互

**定义**：三个及以上条件同时为true时的组合效应。

**示例**：
```
组合: T T T F F F F F F F  (a=T, b=T, c=T)
```

**特点**：组合数爆炸，难以全覆盖，是覆盖率的主要缺口。

---

## 3. 风险权重体系

### 3.1 权重打分规则

基于SQL文本特征打分，不依赖执行计划：

| 特征 | 分值 | 说明 |
|------|------|------|
| 涉及大表 | +3 | users/orders/logs等常见大表 |
| 包含JOIN | +2 | JOIN是性能杀手 |
| SELECT * | +2 | 回表开销 |
| 包含子查询 | +2 | 子查询放大数据量 |
| 包含OR条件 | +1 | OR难以使用索引 |
| 包含ORDER BY | +1 | 排序有成本 |
| 包含GROUP BY | +1 | 分组聚合有成本 |
| LIKE前缀通配 | +2 | LIKE '\%xxx'无法使用索引 |
| 包含函数 | +1 | 函数导致索引失效 |

### 3.2 示例

```xml
<if test="isVIP">
  SELECT * FROM orders o 
  JOIN users u ON o.user_id = u.id
  WHERE o.status = 'pending' 
  ORDER BY o.create_time
</if>
```

| 特征 | 得分 |
|------|------|
| orders大表 | +3 |
| JOIN | +2 |
| SELECT * | +2 |
| ORDER BY | +1 |
| **总计** | **8** |

## 4. 阶梯式采样方案

### 4.1 阶梯设计

采用**贪心+补充**的阶梯式策略，逐步提高覆盖率：

```
┌─────────────────────────────────────────────────────────────────┐
│                     阶梯式采样流程                                │
└─────────────────────────────────────────────────────────────────┘

  ┌─────────────┐
  │ Step 1      │
  │ 单因子覆盖   │ ──→ 覆盖所有单因子效应 (11个)
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ Step 2      │                    ┌──────────────────┐
  │ 高权重两两   │ ──→ 补充覆盖      │ 关键两两交互    │
  │ 交互补充    │     高权重交互对    │ (按权重排序)    │
  └──────┬──────┘                    └────────┬─────────┘
         │                                     │
         ▼                                     │
  ┌─────────────┐                             │
  │ Step 3      │                             │
  │ 高风险三阶   │ ──→ 补充覆盖                │
  │ 交互补充    │     高权重三阶组合           │
  └──────┬──────┘                             │
         │                                     │
         ▼                                     │
  ┌─────────────┐                             │
  │ Step 4      │                             │
  │ 风险得分    │ ──→ 按风险得分                │
  │ 贪心填充    │     贪心选取剩余组合         │
  └──────┬──────┘                             │
         │                                     │
         ▼                                     │
  ┌─────────────┐
  │ 完成        │ ──→ 输出50个组合
  └─────────────┘
```

### 4.2 各阶梯详解

#### Step 1：单因子覆盖（11个）

**保证**：每个条件的T/F状态至少出现一次

```
[1]  T F F F F F F F F F   (a=T)
[2]  F T F F F F F F F F   (b=T)
[3]  F F T F F F F F F F   (c=T)
[4]  F F F T F F F F F F   (d=T)
[5]  F F F F T F F F F F   (e=T)
[6]  F F F F F T F F F F   (f=T)
[7]  F F F F F F T F F F   (g=T)
[8]  F F F F F F F T F F   (h=T)
[9]  F F F F F F F F T F   (i=T)
[10] F F F F F F F F F T   (j=T)
[11] F F F F F F F F F F   (all False)
```

**覆盖率**：
- 单因子效应：100%
- 两两交互：0%
- 三阶交互：0%

---

#### Step 2：高权重两两交互补充（20个）

**策略**：按权重排序，选取top-10对交互覆盖

**假设权重排序**：
```
a+b: 14分 (8+6)
a+c: 13分 (8+5)
b+c: 11分 (6+5)
a+d: 10分 (8+2)
b+d: 8分  (6+2)
...
```

**选取规则**：
- 每对交互生成2个组合：(T,T) 和 (T,F)
- 已有组合不重复添加

```
[12] T T F F F F F F F F   (a=T, b=T) ✓ 新增
[13] T F F F F F F F F F   (a=T, b=F) ✓ 新增 - 但已在Step1
[14] T F T F F F F F F F   (a=T, c=T) ✓ 新增
[15] F T F F F F F F F F   (b=T) ✓ 新增 - 但已在Step1
...
```

**实际新增**：约15-20个（部分已在Step1）

**覆盖率**：
- 单因子效应：100%
- 两两交互：~40%（10对/45对）
- 三阶交互：0%

---

#### Step 3：高风险三阶交互补充（10个）

**策略**：选取权重最高的三阶组合

**Top-3三阶组合**：
```
a+b+c: 19分 (8+6+5)
a+b+d: 13分 (8+6+2)
a+c+d: 12分 (8+5+2)
...
```

**新增组合**：
```
[32] T T T F F F F F F F   (a=T, b=T, c=T) ✓ 新增
[33] T T F T F F F F F F   (a=T, b=T, d=T) ✓ 新增
[34] T F T T F F F F F F   (a=T, c=T, d=T) ✓ 新增
... (再选7个)
```

**覆盖率**：
- 单因子效应：100%
- 两阶交互：~40%
- 三阶交互：~15%（3个三阶组合）

---

#### Step 4：风险得分贪心填充（9个）

**策略**：计算每个候选组合的总风险得分，贪心选取最高的

**候选池**：剩余未选取的高风险组合

**计算公式**：
```
组合风险得分 = Σ(if_true的权重)
```

**示例**：
```
[42] T T T T F F F F F F  (a+b+c+d全开)  得分=21  ← 最高
[43] T T T F F F F F F F  (a+b+c)         得分=19
[44] T T F T T F F F F F  (a+b+d+e)       得分=18
...
```

**选取**：贪心取top-9

---

### 4.3 各阶梯覆盖率汇总

| 阶梯 | 新增组合数 | 累计组合数 | 单因子覆盖 | 两两交互覆盖 | 三阶交互覆盖 |
|------|-----------|------------|------------|--------------|--------------|
| Step1 | 11 | 11 | 100% | 0% | 0% |
| Step2 | 20 | 31 | 100% | ~40% | 0% |
| Step3 | 10 | 41 | 100% | ~40% | ~15% |
| Step4 | 9 | 50 | 100% | ~40% | ~15% |

## 5. 覆盖率分析

### 5.1 理论覆盖率

基于SQL性能问题的分布假设：

| 问题类型 | 占比估算 | 覆盖情况 |
|----------|----------|----------|
| 单因子效应 | 60% | ✅ 100%覆盖 |
| 两两交互 | 25% | 🟡 ~40%覆盖 |
| 三阶交互 | 10% | 🟡 ~15%覆盖 |
| 四阶及以上 | 5% | ❌ 未覆盖 |

**加权覆盖率**：
```
覆盖率 = 60% * 100% + 25% * 40% + 10% * 15% + 5% * 0%
       = 60% + 10% + 1.5% + 0%
       = 71.5%
```

### 5.2 提升覆盖率的方法

| 目标覆盖率 | 需要组合数 | 策略调整 |
|------------|------------|----------|
| 80% | ~70 | 覆盖所有两两交互(45对→90个) |
| 90% | ~120 | 覆盖所有三阶交互 |
| 95% | ~200 | 覆盖所有四阶交互 |
| 99% | ~400 | 覆盖所有五阶交互 |

### 5.3 阶梯扩容策略

如果50个组合不够，可以按需扩容：

```
50 → 70个：补充覆盖所有两两交互
70 → 120个：补充三阶交互
120 → 200个：补充四阶交互
```

## 6. 自适应策略

### 6.1 动态反馈机制

```
┌─────────────────────────────────────────────────────────────────┐
│                       自适应采样流程                              │
└─────────────────────────────────────────────────────────────────┘

  ┌─────────────┐
  │ 初始采样    │
  │ (50个)     │
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ Recognition  │ ──→ 执行计划分析
  │ 阶段        │     检测新问题模式
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ 问题模式    │ ──→ 发现未覆盖的高风险组合?
  │ 检测        │
  └──────┬──────┘
         │
    ┌────┴────┐
    │         │
   YES        NO
    │         │
    ▼         ▼
┌─────────────┐   ┌─────────────┐
│ 补充采样    │   │ 完成        │
│ 回到Recognition│
└─────────────┘   └─────────────┘
```

### 6.2 触发补充的条件

| 条件 | 动作 |
|------|------|
| 发现未覆盖的两两交互 | 补充该对的(T,T)和(T,F)组合 |
| 发现未覆盖的三阶交互 | 补充该三阶组合 |
| 新检测到高风险模式 | 补充相关的高权重组合 |
| 覆盖率低于阈值 | 自动扩容到下一阶梯 |

## 7. 完整算法伪代码

```python
def generate_sampling(ifs: list[IfCondition], budget: int = 50) -> list[Combination]:
    """
    生成采样组合
    
    Args:
        ifs: 10个if条件，每个有weight属性
        budget: 预算上限
    
    Returns:
        组合列表
    """
    combos = []
    covered = set()  # 已覆盖的组合(用tuple表示)
    
    # === Step 1: 单因子覆盖 ===
    # 每个条件的T状态
    for i, if_cond in enumerate(ifs):
        combo = [F] * len(ifs)
        combo[i] = T
        combos.append(combo)
        covered.add(tuple(combo))
    
    # all False
    all_false = [F] * len(ifs)
    combos.append(all_false)
    covered.add(tuple(all_false))
    
    # === Step 2: 高权重两两交互 ===
    pairs = get_top_weighted_pairs(ifs, top_k=10)
    for (i, j) in pairs:
        # (T, T)
        combo_tt = [F] * len(ifs)
        combo_tt[i] = T
        combo_tt[j] = T
        if tuple(combo_tt) not in covered:
            combos.append(combo_tt)
            covered.add(tuple(combo_tt))
        
        # (T, F) - 已在Step1覆盖，跳过
    
    # === Step 3: 高风险三阶交互 ===
    triples = get_top_weighted_triples(ifs, top_k=5)
    for (i, j, k) in triples:
        combo = [F] * len(ifs)
        combo[i] = T
        combo[j] = T
        combo[k] = T
        if tuple(combo) not in covered:
            combos.append(combo)
            covered.add(tuple(combo))
    
    # === Step 4: 贪心填充 ===
    while len(combos) < budget:
        # 计算所有候选组合的风险得分
        candidates = []
        for combo in generate_all_combos(ifs):
            if tuple(combo) not in covered:
                score = sum(if_cond.weight for if_cond, val in zip(ifs, combo) if val == T)
                candidates.append((score, combo))
        
        if not candidates:
            break
        
        # 贪心选取得分最高的
        candidates.sort(reverse=True)
        _, best = candidates[0]
        combos.append(best)
        covered.add(tuple(best))
    
    return combos[:budget]
```

## 8. 配置项设计

```yaml
parse:
  # 分支采样配置
  branch_sampling:
    # 采样预算
    budget: 50
    
    # 是否启用采样（否则全展开）
    enabled: true
    
    # 阶梯配置
    ladder:
      # Step1: 单因子覆盖
      single_factor:
        enabled: true
        force_all_false: true  # 是否强制包含全false组合
      
      # Step2: 两两交互
      pairwise:
        enabled: true
        top_k: 10  # 选取top多少对
      
      # Step3: 三阶交互
      triple:
        enabled: true
        top_k: 5   # 选取top多少三阶组合
      
      # Step4: 贪心填充
      greedy:
        enabled: true
        scoring: "sum_weight"  # sum_weight | min_max
    
    # 自适应配置
    adaptive:
      enabled: true
      coverage_threshold: 0.7  # 覆盖率低于此值时触发扩容
      max_budget: 100         # 最大预算

  # 回退策略（当采样预算大于实际组合数时）
  fallback: "full_expand"  # full_expand | sample_random
```

## 9. 与其他策略的配合

### 9.1 配合Parse阶段剪枝

采样压缩发生在**展开之后**，与以下策略配合：

```
展开后分支列表 (1024个)
        │
        ▼
┌─────────────────┐
│ 死代码识别(A1)   │ ──→ 移除无效分支 → 假设剩余900个
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 条件互斥(A2)    │ ──→ 合并互斥分支 → 假设剩余700个
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 等价合并        │ ──→ 去除重复SQL → 假设剩余500个
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 采样压缩        │ ──→ 贪心选取50个代表
└────────┬────────┘
         │
         ▼
    50个代表组合
```

### 9.2 配合Recognition阶段

采样后的50个组合进入Recognition阶段，基于执行计划再次评估风险，触发自适应补充采样。

## 10. 总结

### 10.1 方案优势

| 优势 | 说明 |
|------|------|
| **渐进覆盖** | 阶梯式设计，保证基础覆盖，逐步提升 |
| **可预测预算** | 固定预算（如50个），避免资源耗尽 |
| **自适应** | 可根据Recognition结果动态补充 |
| **风险导向** | 高权重条件优先覆盖 |
| **可配置** | 各阶梯参数可调 |

### 10.2 覆盖率

| 规模 | 单因子 | 两两交互 | 三阶交互 | 加权覆盖 |
|------|--------|----------|----------|----------|
| 50个 | 100% | ~40% | ~15% | ~72% |
| 70个 | 100% | ~70% | ~20% | ~82% |
| 100个 | 100% | ~90% | ~40% | ~91% |

### 10.3 适用场景

- **10个if条件，预算50个组合**：满足大部分场景，约72%覆盖率
- **更高要求**：可扩容到70-100个组合，覆盖率提升到80-90%
