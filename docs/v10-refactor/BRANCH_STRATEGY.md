# 分支展开与剪枝策略方案

## 1. 背景

SQL优化工具的第二阶段（Parse阶段）负责将MyBatis XML中的动态SQL展开为可执行的分支SQL。然而，动态SQL的嵌套结构可能导致分支数量指数级爆炸，严重影响后续LLM优化阶段的性能。

本方案旨在设计一套**全面展开 + 智能剪枝**的策略，在保证不遗漏高风险SQL的前提下，最大程度减少无效分支的处理开销。

---

## 2. 分支展开策略决策

### 2.1 原有设计：三种展开策略

原有设计提出了三种展开策略：

| 策略 | 描述 |
|------|------|
| **全面展开** | 所有分支都展开，完整覆盖所有条件组合 |
| **两两配对** | 多个动态条件只进行两两组合，减少分支数 |
| **边界值展开** | 只展开边界/极端情况，快速验证 |

### 2.2 决策结论

**结论：只保留"全面展开"策略，淘汰"两两配对"和"边界值展开"。**

### 2.3 淘汰原因

#### 2.3.1 淘汰"两两配对"

**原因**：
1. **覆盖不完整**：两两配对会漏掉高阶条件组合的交叉效应。例如三个条件`A ∧ B ∧ C`，两两配对只覆盖`A∧B`、`B∧C`、`A∧C`，但`A∧B∧C`可能产生完全不同的执行计划
2. **与辅助策略重复**："两两配对"的目的是控制分支数量，但这可以通过"辅助策略层"的预算控制（A14）和优先级分流（A15）更优雅地实现
3. **实现复杂度高**：需要正确处理条件之间的依赖关系，实现难度大且容易出错

**示例**：
```xml
<if test="isVIP">...</if>
<if test="hasCoupon">...</if>
<if test="isNewUser">...</if>
```
- 两两配对：只组合`isVIP∧hasCoupon`、`hasCoupon∧isNewUser`、`isVIP∧isNewUser`
- 遗漏：`isVIP∧hasCoupon∧isNewUser`（超级用户+有券+新用户）

#### 2.3.2 淘汰"边界值展开"

**原因**：
1. **覆盖不完整**：边界值只覆盖"极端情况"，但性能问题往往出现在"普通情况"
2. **需要业务知识**：边界值的定义需要业务人员介入，不适合自动化工具
3. **辅助策略可替代**："边界值"本质上是一种启发式筛选，规则匹配预判（A10）可以更精确地识别需要优化的SQL

**示例**：
```xml
<if test="amount > 10000">大额订单特殊处理</if>
```
- 边界值观点认为：`amount > 10000`是边界，值得展开
- 但实际上：普通金额的查询频率更高，累计性能影响更大

## 3. 辅助策略体系

### 3.1 策略全景图

| 策略 | 名称 | 介入阶段 | 状态 |
|------|------|----------|------|
| A1 | 死代码分支识别 | Parse | ✅ 选用 |
| A2 | 条件互斥分析 | Parse | ✅ 选用 |
| A3 | 条件依赖裁剪 | Parse | ✅ 选用 |
| A4 | 相似分支聚类 | Recognition | ✅ 选用 |
| A5 | 参数化程度评估 | Recognition | ✅ 选用（修正） |
| A6 | 表规模感知 | Recognition | ✅ 选用 |
| A7 | 索引覆盖识别 | Recognition | ✅ 选用 |
| A8 | 执行频率预估 | - | ❌ 淘汰 |
| A9 | 历史优化收益 | - | ⚠️ 长期 |
| A10 | 规则匹配预判 | Optimize | ✅ 选用 |
| A11 | 优化收益预估 | - | ❌ 淘汰 |
| A12 | LLM优化衰减 | - | ⚠️ 长期 |
| A13 | 分支血缘追踪 | - | ⚠️ 长期 |
| A14 | 展开预算控制 | Parse | ✅ 选用 |
| A15 | 优先级标注 | Parse | ✅ 选用 |
| R2 | 问题模式识别 | Recognition | ✅ 选用 |
| - | 等价分支合并 | Parse | ✅ 选用 |

### 3.2 淘汰策略详细说明

#### 3.2.1 淘汰 A8 执行频率预估

**原因**：静态分析无法获取业务执行频率数据。执行频率属于运行时业务指标，需要埋点或监控系统的支持。

**结论**：暂不考虑，后续可通过集成业务监控数据实现。

#### 3.2.2 淘汰 A11 优化收益预估

**原因**：基于LLM自评的优化收益不可靠。LLM声称"提升30%"缺乏客观验证，执行计划对比（优化前cost vs 优化后cost）需要两次Recognition阶段，成本过高。

**替代方案**：通过规则预判（A10）直接识别已知的高风险SQL模式，不依赖收益预估。

#### 3.2.3 长期特性 A9、A12、A13

| 策略 | 说明 | 前提条件 |
|------|------|----------|
| A9 历史优化收益 | 复用已有优化结论 | 需要建立优化历史缓存系统 |
| A12 LLM优化衰减 | 检测重复优化 | 需要历史缓存 + 效果追踪 |
| A13 分支血缘追踪 | 优化问题归因 | 需要完整的执行链路追踪 |

这些特性依赖基础设施，建议作为后续迭代的高级特性。

## 4. 选用策略详解

### 4.1 Parse阶段策略

#### 4.1.1 A1 死代码分支识别

**问题场景**：
```xml
<if test="1 = 0">-- 永远为false
  SELECT * FROM users WHERE id = 1
</if>
<if test="status != status">-- 永远为false
  AND status = 'deleted'
</if>
```

**识别方法**：静态分析条件表达式，检测永真/永假模式：
- `1 = 0`、`0 = 1`
- `a != a`
- `a AND NOT a`
- 包含`false`、`null`的比较

**处理**：直接丢弃，不加入分支列表

**收益**：零成本，字符串处理即可，移除无效分支

---

#### 4.1.2 A2 条件互斥分析

**问题场景**：
```xml
<choose>
  <when test="type == 'A'">...A...</when>
  <when test="type == 'B'">...B...</when>
  <when test="type == 'C'">...C...</when>
  <otherwise>...default...</otherwise>
</choose>
```

**识别方法**：`choose/when`结构天然互斥，同一时刻只有一个分支生效。

**处理**：
- 将`when`和`otherwise`作为独立候选分支
- 最终只选择**一个**分支（由运行时条件决定）
- 注意：互斥分析后不需要展开成所有组合

**收益**：原本可能展开成N个分支（两两组合），互斥分析后只需处理实际会生效的分支

---

#### 4.1.3 A3 条件依赖裁剪

**问题场景**：
```xml
<if test="isVIP">
  <if test="discount != null and discount > 0">
    AND discount_amount > 0
  </if>
</if>
```

**识别方法**：内层`if`的test引用了外层`if`的变量：
- 外层条件`isVIP`为false时，内层`discount != null`无论真假都无效
- 可裁剪内层条件的展开

**处理**：只在外层条件为true时，才展开内层分支

**收益**：减少无效组合，例如`isVIP=false`时不需要展开内层的各种discount情况

---

#### 4.1.4 等价分支合并

**问题场景**：
```xml
<if test="status != null">AND status = 'active'</if>
<otherwise>AND status = 'active'</otherwise>
```

展开后可能产生完全相同的SQL。

**识别方法**：展开后比较SQL文本，完全相同则合并

**处理**：
```json
{
  "path_id": "if_0|otherwise",
  "condition": "status != null | otherwise",
  "expanded_sql": "SELECT * FROM orders WHERE status = 'active'"
}
```

**收益**：相同SQL只优化一次，后续阶段工作量减半

---

#### 4.1.5 A14 展开预算控制

**兜底机制**，防止极端情况导致分支爆炸。

**触发条件**：当分支数量超过预设阈值（如100）时

**处理策略**：
1. **按优先级保留**：优先保留高风险分支（复杂SQL、涉及大表）
2. **随机采样+高优先级保证**：保留80%高优先级，20%随机采样
3. **警告日志**：记录被裁剪的分支信息

**配置项**：
```yaml
parse:
  branch_budget:
    max_branches: 100
    priority_threshold: "HIGH"
    fallback_strategy: "sample"
```

**收益**：防止系统资源耗尽，是极端情况的最后防线

---

#### 4.1.6 A15 优先级标注

为每个分支打上优先级标签，供后续阶段分流。

**评分维度**：

| 维度 | 高分标准（→ P0） | 低分标准（→ P2） |
|------|------------------|------------------|
| SQL复杂度 | >5个JOIN，子查询>2层 | 无JOIN，单表 |
| 硬编码程度 | 大量硬编码值 | 参数化 |
| 条件嵌套深度 | >3层嵌套 | 无嵌套 |

**优先级定义**：

| 优先级 | 定义 | 处理方式 |
|--------|------|----------|
| **P0** | 高风险 + 高成本 | 立即优化，优先调LLM |
| **P1** | 高风险 OR 高成本 | 批量优化 |
| **P2** | 低风险 + 低成本 | 可选优化，或通过规则直接处理 |

**收益**：后续阶段可以根据优先级分流，避免在低价值SQL上浪费LLM调用

---

### 4.2 Recognition阶段策略

Recognition阶段基于执行计划补充风险信息。

#### 4.2.1 R2 问题模式识别

从执行计划中识别已知的性能问题模式：

| 问题模式 | 权重 | 说明 |
|----------|------|------|
| Seq Scan on large_table | 10 | 全表扫描大表 |
| Nested Loop | 5 | 嵌套循环 |
| Hash Join (large) | 3 | 大表哈希连接 |
| Sort | 2 | 额外排序操作 |
| Bitmap Heap Scan | 2 | 位图扫描 |

**处理**：累加权重，高权重分支优先优化

**示例**：
```
执行计划：
  → Seq Scan on orders (cost=1500)
  → Nested Loop (cost=2000)
  → Sort (cost=100)
问题模式得分：10 + 5 + 2 = 17 → P0
```

---

#### 4.2.2 A4 相似分支聚类

**注意**：相似分支聚类必须在Recognition阶段基于**执行计划**判断，不能仅基于SQL文本。

**问题**：
```sql
-- 分支A（food订单，可能很大）
SELECT * FROM orders WHERE status = 'pending' AND type = 'food'

-- 分支B（electronic订单，可能较小）
SELECT * FROM orders WHERE status = 'pending' AND type = 'electronic'
```

如果仅比较SQL文本，两者结构完全相同（相似）。但数据分布不同，优化方向可能不同。

**识别方法**：
1. 执行EXPLAIN获取两个分支的执行计划
2. 比较执行计划的：
   - 预估扫描行数
   - 是否走索引
   - join方式
   - 排序操作
3. 如果执行计划高度相似（扫描行数在同一数量级、索引使用情况相同），则归为一类

**处理**：同一聚类的分支共享优化结论，或选择"最差情况"的分支进行优化

**收益**：避免对大量结构相似但执行计划相同的分支重复优化

---

#### 4.2.3 A5 参数化程度评估（修正）

**修正理解**：参数化程度不决定"是否优化"，而决定"优化结论的复用度"。

**评估方法**：
```sql
SELECT COUNT(*) FROM orders WHERE status = 'pending'  -- 硬编码多
SELECT COUNT(*) FROM orders WHERE status = ?         -- 参数化
```

**指标**：硬编码值数量 / 总条件数

**作用**：
- 高参数化 → 优化结论可泛化到同类SQL
- 低参数化 → 优化结论仅适用于当前SQL

**收益**：高参数化SQL优先优化，收益更大

---

#### 4.2.4 A6 表规模感知

Init阶段已获取表元数据（行数、索引信息）。

**评估方法**：
- 大表（>100万行）+ 全表扫描 → P0
- 中表（1万-100万行）→ P1
- 小表（<1万行）→ P2

**注意**：小表即使全表扫描也可能很快，优化收益有限

---

#### 4.2.5 A7 索引覆盖识别

从执行计划中判断查询是否已被索引覆盖。

**识别方法**：
```sql
-- 覆盖索引（Index Only Scan）
SELECT id, name FROM users WHERE id = 1  -- id是主键

-- 非覆盖索引（需要回表）
SELECT * FROM users WHERE id = 1  -- *包含非索引字段
```

**处理**：
- 如果执行计划显示`Index Only Scan`且查询列都在索引中 → P2（优化空间有限）
- 如果需要回表或走Seq Scan → 保持原有优先级

---

### 4.3 Optimize阶段策略

#### 4.3.1 A10 规则匹配预判

通过正则/规则直接识别高风险SQL，不调LLM。

**规则示例**：

| 规则 | 模式 | 优化建议 |
|------|------|----------|
| SELECT * | `SELECT \*` | 明确列，避免回表 |
| IN子句过多 | `WHERE id IN (...20+...)` | 拆分为多次查询或JOIN临时表 |
| 不使用绑定变量 | `WHERE status = 'hardcoded'` | 改用参数绑定 |
| 隐式类型转换 | `WHERE num = '123'` | 类型匹配 |

**处理**：命中规则的SQL，直接应用优化建议，跳过LLM调用

**收益**：减少LLM调用，对简单明确的问题快速响应

## 5. 整体架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Parse 阶段                                  │
├─────────────────────────────────────────────────────────────────────┤
│  输入：MyBatis XML SQL单元                                           │
│                                                                     │
│  处理：                                                              │
│  1. 全面展开所有动态标签（if/choose/when/otherwise）                   │
│  2. A1 死代码识别 → 移除永真/永假条件                                  │
│  3. A2 互斥分析 → choose/when只保留一个生效分支                        │
│  4. A3 依赖裁剪 → 外层false时内层不展开                               │
│  5. 等价合并 → 文本相同SQL合并                                        │
│  6. A14 预算控制 → 超阈值时按优先级裁剪                                │
│  7. A15 优先级标注 → 打P0/P1/P2标签                                   │
│                                                                     │
│  输出：带优先级标签的分支列表                                          │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│                      Recognition 阶段                               │
├─────────────────────────────────────────────────────────────────────┤
│  输入：Parse阶段输出的分支列表                                         │
│                                                                     │
│  处理：                                                              │
│  1. 执行EXPLAIN获取执行计划                                           │
│  2. R2 问题模式识别 → 累加问题模式得分                                │
│  3. A6 表规模感知 → 结合表行数调整优先级                              │
│  4. A7 索引覆盖识别 → 覆盖索引的分支降级                              │
│  5. A4 相似聚类 → 执行计划相似的分支归为一类                           │
│  6. 综合评分 → 更新优先级（P0/P1/P2）                                  │
│                                                                     │
│  输出：带执行计划和综合优先级的分支列表                                 │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│                       Optimize 阶段                                 │
├─────────────────────────────────────────────────────────────────────┤
│  输入：带优先级的分支列表                                              │
│                                                                     │
│  处理（按优先级分流）：                                                │
│                                                                     │
│  P0分支：                                                            │
│    → 直接调LLM优化（最高优先级）                                       │
│                                                                     │
│  P1分支：                                                            │
│    → 批量调LLM优化                                                   │
│                                                                     │
│  P2分支：                                                            │
│    → A10 规则预判（命中规则 → 直接应用优化）                            │
│    → 未命中规则 → 可选优化或跳过                                      │
│                                                                     │
│  输出：优化建议                                                       │
└─────────────────────────────────────────────────────────────────────┘
```

## 6. 配置项设计

```yaml
parse:
  # 展开策略
  expand_strategy: "full"  # 只支持full，淘汰partial/boundary
  
  # 预算控制
  branch_budget:
    max_branches: 100
    fallback_strategy: "priority_sample"  # priority_sample | random | all
  
  # 优先级权重
  priority:
    complexity_weight: 0.4
    cost_weight: 0.3
    risk_pattern_weight: 0.3
    
recognition:
  # 问题模式权重
  problem_patterns:
    seq_scan: 10
    nested_loop: 5
    hash_join_large: 3
    sort: 2
    bitmap_heap: 2
  
  # 表规模阈值
  table_size:
    large: 1000000   # >100万行
    medium: 10000    # >1万行
    # <1万行视为小表

optimize:
  # LLM调用策略
  llm_bypass:
    enabled: true
    rules:
      - pattern: "SELECT \\*"
        action: "rewrite_select"
      - pattern: "WHERE id IN \\(\\d+(,\\d+){20,}\\)"
        action: "split_in_clause"
```

## 7. 总结

### 7.1 淘汰项

| 策略 | 淘汰原因 |
|------|----------|
| 两两配对展开 | 覆盖不完整，实现复杂，与辅助策略功能重复 |
| 边界值展开 | 需要业务知识，覆盖不完整 |
| A8 执行频率预估 | 静态分析无法获取运行时数据 |
| A11 优化收益预估 | LLM自评不可靠，无客观验证方式 |

### 7.2 选用项

| 阶段 | 策略 | 核心价值 |
|------|------|----------|
| Parse | A1 死代码识别 | 移除无效分支 |
| Parse | A2 条件互斥分析 | 利用choose语义减少分支 |
| Parse | A3 条件依赖裁剪 | 减少无效组合 |
| Parse | 等价分支合并 | 去重，减少重复工作 |
| Parse | A14 预算控制 | 兜底，防止指数爆炸 |
| Parse | A15 优先级标注 | 为后续分流提供依据 |
| Recognition | R2 问题模式识别 | 基于执行计划识别高风险 |
| Recognition | A4 相似分支聚类 | 避免重复优化相似SQL |
| Recognition | A5 参数化评估 | 评估优化结论复用度 |
| Recognition | A6 表规模感知 | 结合数据量判断优化价值 |
| Recognition | A7 索引覆盖识别 | 识别已优化的SQL |
| Optimize | A10 规则匹配预判 | 简单SQL不调LLM |

### 7.3 长期特性

| 策略 | 说明 | 前提 |
|------|------|------|
| A9 历史优化收益 | 缓存复用 | 需要历史缓存系统 |
| A12 LLM优化衰减 | 避免重复优化 | 需要效果追踪 |
| A13 分支血缘追踪 | 优化归因debug | 需要完整执行链路 |
